## XI. THE ITERATION RITUAL (wiki-coupled execution)

Execution is bookended by the knowledge wiki, so each iteration starts ahead of the last.

- **Start every iteration by READING.** Load the wiki (pull from GitHub when reachable, else it
  is uploaded) and read the pages relevant to the task — value pages, the relevant bug-class
  pages, prior prompt shapes — BEFORE writing anything. This is §I (read before you write)
  extended across sessions.
- **End every iteration by WRITING BACK.** Hand back the changed pages as a diff: new/updated
  value pages, a new instance on a class page (and a sweep proposal when instances reach 3),
  any prompt shape that worked, the `index.md` and `log.md` updates. Gary commits and pushes.
- **The model is the maintainer, not the persistence layer.** It maintains the wiki *during* a
  session; it does not carry the wiki between sessions. Freshness comes from the repo + this
  ritual, not from model memory. If a load is skipped, say so — do not reconstruct from memory
  and present it as canon.
- **Gates (unchanged spine):** the model auto-owns bookkeeping only. Every decision routes to
  **Gary**. Compliance values stay **❓** until **Brian** certifies *through Gary* — the model
  never writes ✅/❌ itself and never contacts Brian. **Promoting** a lesson INTO CLAUDE.md or
  RULES (raising the bar) is **Gary-only**; per the RQGM note that act must stay human.
- **Canon rule:** only verified knowledge enters as fact — primary source (statute/O.Reg/HTA)
  **and** confirmed secondary corroboration; statute/primary wins on conflict (RULES §10).
  Unverified material goes to `pages/staging/`, never silently into a page.
